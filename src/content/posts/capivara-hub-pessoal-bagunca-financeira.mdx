No meio de junho, enquanto o Dogwalk engrenava e o Arachne começava a tomar forma, uma verdade incômoda bateu na porta: eu tinha perdido o controle dos meus projetos.

Não era uma crise — era bagunça. Senhas de serviços, chaves de API, endpoints de deploy, variáveis de ambiente, configurações de infraestrutura — tudo espalhado em arquivos .env, anotações soltas, e documentos perdidos. Cada projeto tinha seu próprio ecossistema de segredos e configurações, e eu não tinha uma visão unificada de nada.

## O estalo (de novo)

13 de junho. Eu estava caçando uma chave de API no terminal quando pensei: *"Isso é ridículo. Eu construo sistemas distribuídos, por que não construir um hub central pro meu próprio ecossistema?"*

O nome veio antes do código. Capivara — o animal calmo que todo mundo gosta, mas que também sabe se impor. Queria um hub pessoal que fosse acolhedor, mas sério. Que organizasse minha vida digital sem parecer um sistema corporativo.

## Primeiro commit

O primeiro commit foi documental: `docs: descricao detalhada do projeto no AGENTS.md`. Antes de escrever uma linha de código, documentei exatamente o que o Capivara precisava ser.

```markdown
# Capivara
Hub pessoal seguro para gerenciamento de projetos, segredos,
credenciais e acesso centralizado ao ecossistema.

Stack: FastAPI + React 19 + SQLite

Funcionalidades planejadas:
- Gerenciamento centralizado de credenciais e chaves de API
- Painel de controle de projetos (status, health, métricas)
- Visão consolidada de serviços, domínios e endpoints
- Modo escuro porque sim
```

## O primeiro endpoint

Horas depois veio o código de verdade. O endpoint mais simples possível — um `/health` que serviria de termômetro pro sistema:

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse

app = FastAPI(title="Capivara", version="0.1.0")


@app.get("/health")
async def health():
    return JSONResponse(
        content={"status": "ok", "version": "0.1.0", "project": "capivara"},
        status_code=200,
    )
```

Simples, quase bobo. Mas todo grande sistema começa com um endpoint que responde "ok". O Capivara estava vivo.

## De "Mission Control" a "Capivara"

Antes do nome final, o projeto se chamava "Mission Control" — uma referência a centro de controle de missões espaciais. O nome era legal, mas não tinha personalidade. "Capivara" veio da vontade de ter algo mais brasileiro, mais próximo, menos genérico.

E fez sentido. Capivaras são animais gregários, que vivem em grupos, são adaptáveis — exatamente o que eu queria pro hub que centralizaria todos os meus projetos.

O Capivara que nasceu em junho da necessidade de controle se tornaria, nos meses seguintes, o cérebro administrativo de todo o ecossistema Samuel. Mas isso é história pra outro post.
