A primeira versão dos testes E2E do Dogwalk foi o ponto de partida — um `auth.spec.ts` simples que verificava se a página de login carregava. O post anterior contou essa história. Mas o que veio depois foi uma reestruturação completa.

Não bastava ter testes. Precisávamos de uma **arquitetura de testes** que:

- Rodasse em CI sem depender de ambiente externo
- Cobrisse fluxos multi-perfil (tutor e walker autenticados)
- Fosse rápida o suficiente pra dar feedback em segundos
- Suportasse mocks de API reais, não stubs genéricos

Este post é um mergulho técnico nessa arquitetura.

## O core da estratégia: injeção de sessão mock

O pulo do gato foi o helper `setupTutorSession()` / `setupWalkerSession()`. Ele faz **três coisas** em uma chamada:

```javascript
// tests/e2e/helpers/auth.js

export async function setupTutorSession(page) {
  const session = makeSession('tutor')
  await addAuthInitScript(page, session)
  await setupNetworkMock(page, { user: MOCK_USERS.tutor, authenticated: true })
}
```

**1. Injeta sessão via `addInitScript`**

O `AuthContext` do frontend tem um backdoor: se `window.__E2E_SESSION` existir, ele usa direto — zero chamadas de API:

```javascript
async function addAuthInitScript(page, session) {
  await page.addInitScript((sess) => {
    window.__E2E_SESSION = sess
    localStorage.setItem('dw_access_token', sess.access_token || 'e2e-mock-token')
    localStorage.setItem('dw_refresh_token', 'e2e-mock-refresh')
    if (sess.user) {
      localStorage.setItem('dw_user', JSON.stringify(sess.user))
    }
  }, session)
}
```

Isso significa que o teste não depende de banco, servidor de auth, ou Turnstile. O componente acorda já autenticado.

**2. Mocka a API real via `page.route()`**

O network mock intercepta chamadas para **20 prefixos de rota** (`/auth/`, `/profiles/`, `/pets`, `/bookings`, `/walks`, `/wallet`, `/chat`, etc.) e retorna dados controlados.

Para cada requisição, o dispatcher roteia pro handler certo:

```javascript
await page.route((url) => apiPathPrefixes.some(p => url.pathname.startsWith(p)),
  async (route) => {
    const path = new URL(route.request().url()).pathname

    switch (path) {
      case '/auth/me':
        return route.fulfill({ status: 200, body: JSON.stringify(makeMeResponse(user)) })
      case '/auth/login':
      case '/auth/register':
        return route.fulfill({ status: 200, body: JSON.stringify({
          access_token: 'e2e-mock-token',
          user: { id: user.id, email: user.email, role: user.role },
        })})
      case '/profiles/walkers':
        return route.fulfill({ status: 200, body: JSON.stringify(MOCK_WALKERS) })
      // ... +15 rotas
    }
  }
)
```

**3. Dados mock realistas**

Os mocks não são placeholders genéricos — representam dados de verdade que o frontend espera:

```javascript
const MOCK_USERS = {
  tutor: { id: 'e2e-tutor-id', email: 'tutor@test.patapass', role: 'tutor' },
  walker: { id: 'e2e-walker-id', email: 'walker@test.patapass', role: 'walker' },
}

const MOCK_PETS = [
  { id: 'mock-pet-001', name: 'Rex', species: 'dog', breed: 'Labrador', weight: 28.5 },
  { id: 'mock-pet-002', name: 'Luna', species: 'dog', breed: 'Golden Retriever', weight: 22.0 },
]

const MOCK_WALKERS = [
  { id: 'mock-walker-001', name: 'Maria Passeadora', rating: 4.8, price_per_walk: 35, city: 'São Paulo' },
  { id: 'mock-walker-002', name: 'Carlos PetLover', rating: 4.5, price_per_walk: 25, city: 'São Paulo' },

```

## A configuração do Playwright

A `playwright.config.js` foi projetada pra rodar bem tanto local quanto em CI:

```javascript
export default defineConfig({
  testDir: '.',
  testMatch: ['tests/e2e/**/*.spec.{js,ts}'],
  timeout: 30_000,
  expect: { timeout: 10_000 },
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 2 : undefined,

  use: {
    baseURL: process.env.PLAYWRIGHT_TEST_URL || 'http://localhost:5173',
    headless: true,
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
    locale: 'pt-BR',
    timezoneId: 'America/Sao_Paulo',
  },
  // chromium apenas — suficiente pra regressão
})
```

Detalhes que fizeram diferença:

- **`trace: 'on-first-retry'`** — não polui com trace em testes que passam, mas captura tudo quando flaky
- **`video: 'retain-on-failure'`** — ver o vídeo do teste falhando é 10x mais rápido que debugar log
- **`locale: 'pt-BR'`** — os componentes têm texto em português e validação de CPF/CEP
- **`baseURL` configurável** — `PLAYWRIGHT_TEST_URL` permite testar preview deploy antes de ir pra produção

## Pipeline CI com Woodpecker

O CI tem **6 etapas encadeadas**, e E2E só roda se build passar:

```yaml
when:
  branch: [master]
  event: [push, pull_request, manual]

steps:
  install → lint → test (unit) → build → e2e → deploy
```

A etapa de E2E levanta o preview server, instala o Chromium, e executa:

```yaml
e2e:
  commands:
    - corepack enable && corepack prepare pnpm@9 --activate
    - npx playwright install chromium --with-deps 2>/dev/null || true
    - npx vite preview --port 5173 --host 127.0.0.1 &
    - sleep 3
    - PLAYWRIGHT_TEST_URL=http://127.0.0.1:5173 bash scripts/run-e2e-ci.sh 2>&1 || true
  depends_on: [build]
```

## Wrapper CI com dupla estratégia de reporter

O `scripts/run-e2e-ci.sh` executa os testes **duas vezes**: uma com reporter `list` pra saída legível, outra com `json` pra análise.

```bash
# Pass 1: saída humana
npx playwright test "${ARGS[@]}" || true

# Pass 2: JSON report pra análise
npx playwright test "${ARGS[@]}" --reporter=json 2>/dev/null > "$JSON_FILE" || true

# Processa com Python
python3 -c "
import json
d = json.load(open('$JSON_FILE'))
s = d['stats']
print(f'✅ {s[\"expected\"]}  ❌ {s[\"unexpected\"]}  ⏭  {s[\"skipped\"]}  ⏱  {s[\"duration\"]/1000:.0f}s')
"

# Registra no histórico de flaky
python3 scripts/track-flaky.py "$JSON_FILE"

# Notifica no Telegram
bash scripts/notify-e2e.sh
```

O JSON report também alimenta o histórico de flaky (salvo em `test-results/flaky-history.csv`), que permite rastrear quais testes oscilam entre verde e vermelho ao longo do tempo.

## Lições aprendidas

### 1. Mock de API é melhor que fixture de banco

No começo os E2E usavam banco de testes real. Cada execução precisava de seed, teardown, e isolamento. Com `page.route()`, o teste é deterministico: dados controlados, zero efeito colateral, roda em paralelo.

### 2. addInitScript > page.evaluate()

Injetar sessão antes do JS do app carregar evita race condition. Se o React tentar chamar `/auth/me` antes do mock estar pronto, o teste quebra. `addInitScript` executa antes de qualquer script da página.

### 3. Duas execuções são melhores que um reporter híbrido

Playwright permite múltiplos reporters, mas o JSON reporter só escreve o arquivo no final. Executar duas vezes (uma pra humano, uma pra máquina) é mais simples que configurar reporter customizado.

### 4. CI com retry + vídeo = produtividade

2 retries em CI + vídeo em falha reduziu o tempo de debugging de flaky tests de ~20min para ~2min. O vídeo mostra exatamente o que o navegador viu antes de falhar.

### 5. Testes E2E melhoram o design do código

Saber que algo vai ser testado força componentes mais modulares. O `AuthContext` precisou de um backdoor (`__E2E_SESSION`) que acabou simplificando a lógica de init. O network dispatcher mock inspirou a refatoração do API client pra usar um interceptor centralizado.
